# 이슈 기반 개발 사이클 (Issue-Driven Development)

`.agents/workflows/issue-driven-development.yaml`에 대한 사람이 읽을 수 있는 가이드입니다.

## 개요

upstream 기반 프로젝트(예: Bazzite 위의 Naia OS)에서 **기능 단위 작업의 기본 워크플로우**입니다. 사용자 개입을 앞단에 집중하고, 이후 AI가 품질 게이트와 함께 자율 실행합니다.

### 적용 범위
- **이 워크플로우 (기본값)**: 신규 기능 개발, 기능 단위 버그 수정, 주변 코드 품질이 불확실한 작업
- **development-cycle.yaml**: 기능 변경이 아닌 단순 변경 지시 (오타, 설정값, 격리된 한 줄 수정)

## 핵심 원칙

| 원칙 | 규칙 | 안티패턴 |
|------|------|----------|
| **원본 분석 먼저** | 코드 작성 전 upstream 구현을 완전히 이해 | upstream 읽지 않고 추측으로 설계 |
| **최소 수정** | upstream 흐름을 따르고, 커스텀은 최소 변경으로 | upstream이 이미 제공하는 걸 다시 만듦 |
| **추측 금지** | 실제 코드를 읽고 판단 | 문서나 이름에서 동작을 추측 |
| **땜질 금지** | 올바른 경로인지 먼저 확인 후 수정 | 증상에 패치 5개 쌓기 (근본 원인 1개면 해결) |
| **자의적 개발 금지** | 개선안은 사용자에게 제안 | "이게 더 낫겠다" 판단 후 독단 구현 |
| **동작 보존** | 동작하는 코드를 '개선'하면서 깨뜨리지 않음 | 동작하는 흐름을 '더 나은' 방식으로 전환하다 크래시 |
| **발견 프로토콜** | 예상치 못한 문제 발견 시, 먼저 분류한 후 행동 | 발견한 문제를 분류 없이 즉시 수정. 증상에 패치 쌓기 |

### 발견 프로토콜 (Discovery Protocol)

작업 중 예상치 못한 문제를 발견했을 때, **행동하기 전에 분류**한다.

| 등급 | 설명 | 행동 |
|------|------|------|
| **T0 (정보)** | 예상되는 마찰 (포트 사용 중, 의존성 누락) | 자동 수정, 조용히 |
| **T1 (범위 내)** | 현재 작업을 차단, plan 범위 내 수정 가능 | 자동 수정, progress에 기록 |
| **T2 (범위 밖)** | 현재 작업 범위 밖의 문제 | 사용자에게 보고, 수정하지 않음 |
| **T3 (공유 자원)** | git push, 배포, 공유 DB | 항상 사용자 확인 필요 |
| **T4 (보안)** | 인증정보, 데이터 손실 | 즉시 보고, 절대 자동 수정 금지 |

**분류 기준**: (1) 현재 승인된 plan 범위 내인가? (2) 공유 자원에 영향을 주는가?
- (2)에 YES → T3. (1)에 NO → T2. 그 외 T0 또는 T1.

**모드**:
- **supervised** (기본): T0/T1 자동 수정. T2 이상은 보고 후 정지.
- **auto**: 사용자가 명시적으로 부여 ('자동으로 해', 'auto mode'). T0/T1/T2 자동 수정(로그 포함). T3/T4는 여전히 확인 필요.
  - TTL: 60분 또는 작업/페이즈 변경 시까지
  - 취소: '멈춰', gate 페이즈 진입, TTL 만료, 작업 변경

**빠른 수정 가드**: 5분 내 3회 이상 수정 → 일시 정지하고 근본 원인 확인

## 전체 흐름

```
── 사용자 개입 구간 ──────────────────────────

ISSUE          GitHub Issue 생성/확인
  |
UNDERSTAND     이슈 이해 확인 → 사용자 승인 (gate)
  |
SCOPE          조사 범위/깊이 제안 → 사용자 승인 (gate)
  |
INVESTIGATE    코드 조사 (반복, 근거 필수)
  |
PLAN           구현 계획 → 사용자 승인 (gate)
  |              (Phase를 테스트 가능 단위로 분할)
── AI 자율 실행 구간 ─────────────────────────

BUILD             구현 (Phase별)
  |
REVIEW (per-phase) Phase별 반복 코드리뷰 + 테스트 (2회 연속 클린)
  |
REVIEW (전체)     모든 Phase 완료 후 전체 반복 코드리뷰 (2회 연속 클린)
  |                   ↑
  |             실패 시 원인별 되돌림
E2E TEST ──── (실패 → BUILD/PLAN/INVESTIGATE)
  |
POST-TEST REVIEW  테스트 후 재리뷰 (반복, 2회 연속 클린 패스)
  |
SYNC              컨텍스트 동기화 + 시행착오 반영 → 사용자 확인 (gate)
  |
SYNC VERIFY       미러 정확성 검증 (반복, 2회 연속 클린 패스)
  |
REPORT            결과 요약
  |
COMMIT            커밋 → PR → Issue 링크
  |
CLOSE             교훈 재검토 + 완료보고 → 사용자 확인 (gate)
```

## 필수 게이트 (5개)

이 단계에서는 사용자 확인 없이 다음으로 진행할 수 없습니다.

1. **UNDERSTAND**: "제가 이해한 이슈가 맞는지, 빠진 가정이 있는지 확인해주세요"
2. **SCOPE**: "이 범위/깊이로 조사하겠습니다, 맞나요?"
3. **PLAN**: "이 계획으로 구현하겠습니다, 진행할까요?"
4. **SYNC**: "컨텍스트 업데이트 내용을 확인해주세요"
5. **CLOSE**: "완료보고서입니다. 이슈를 닫아도 될까요?"

## UNDERSTAND 상세

이슈 이해를 3개 차원으로 구조화하고, 각 차원의 미해결 가정을 먼저 목록화한 뒤 사용자에게 확인한다.

| 차원 | 질문 |
|------|------|
| **Goal** | 이게 해결되면 무엇이 달라지는가? (관찰 가능한 결과) |
| **Constraints** | 건드리면 안 되는 것은 무엇인가? |
| **Criteria** | 성공을 어떻게 검증하는가? (구체적이고 확인 가능한 방법) |

## SCOPE 상세

### 이슈 유형별 범위 조정 (issue_type_calibration)

범위를 제안하기 전에 이슈 유형을 판단하고 기본 자세를 결정한다.

| 유형 | 감지 신호 | 기본 범위 | 이유 |
|------|----------|----------|------|
| **GREENFIELD** | "신규 기능", "추가", "구현", "생성" | L1 + L2 + L3(upstream) | 미지의 영역 — 설계 전에 upstream 먼저 읽기 |
| **EXPANSION** | "확장", "지원 추가", "기존 기능에 X도" | L1 + L2 (직접 호출/피호출) | 기존 패턴 재사용 — 통합 지점만 확인 |
| **BUG_FIX** | "수정", "안 됨", "오류", "크래시", "회귀" | 좁게 시작 (L1 only), 근본 원인 못 찾으면 확장 | 증상 패치 위험 — 먼저 근본 원인 위치 특정 |
| **REFACTOR** | "리팩터", "정리", "이동", "이름 변경" | SELECTIVE — L1 + 영향 받는 호출 지점만 | 범위 확장 위험 — 구조 변경 중 로직 절대 건드리지 않음 |

> **EMERGENCY** (서비스 장애, 보안 침해) → IDD 아님. `development-cycle.yaml` 사용.

## INVESTIGATE 상세

### 범위 (Scope)
- **L1**: 직접 수정 대상 파일
- **L2**: L1의 의존성 (import, 호출하는/호출되는 함수)
- **L3**: upstream 원본 (동일 기능의 upstream 구현)

### 깊이 (Depth)
- **Surface**: 함수 시그니처, 설정값, 인터페이스만
- **Flow**: 실행 흐름 추적 (호출 체인, 실행 순서)
- **Internal**: 라인 단위 구현 로직 이해

### 규칙
- 첫 번째 버그 발견에서 멈추지 않음 — 범위 내 코드 전체 완독
- 모든 발견 사항에 코드 위치(파일:라인) 인용 필수
- "~인 것 같습니다" 금지 — 근거 있는 발견만
- 문서보다 실제 코드 우선
- CI/더미 코드와 프로덕션 코드 구분
- **2회 연속 추가 발견이 없을 때까지 반복**

### Shadow Path Tracing — 4가지 실행 경로 추적

행복 경로(happy path)만 추적하면 실제 문제를 놓칩니다. 아래 4가지 경로를 **반드시** 모두 추적합니다.

| 경로 | 설명 | 예시 |
|------|------|------|
| **happy_path** | 정상 사용자 흐름 | 로그인 → 데이터 조회 → 응답 |
| **nil_path** | 예상 입력이 없는 경우 (missing/undefined) | null, undefined, 키 누락 |
| **empty_path** | 길이 0의 유효한 입력 | 빈 문자열, 빈 배열, 0, 빈 객체 |
| **error_path** | upstream 호출 실패 시 동작 | catch 블록, fallback 값, 타임아웃, 네트워크 에러 |

> 추가로 확인: **edge_path** (경계 입력 — 최대 길이, 동시 요청), **security_path** (신뢰할 수 없는 입력 — HTTP body/header, env, 파일)

## REVIEW 체크리스트

| 항목 | 확인 사항 |
|------|----------|
| **Plan 충족** | plan의 각 항목이 구현되었는가? |
| **Issue 해결** | 원래 이슈가 실제로 해결되는가? |
| **범위 준수** | plan에 없는 변경이 섞여 있지 않은가? |
| **기존 동작 보존** | 동작하던 코드를 깨뜨리지 않았는가? |
| **upstream 패턴** | upstream 방식을 따르는가, 자체 방식을 만들지 않았는가? |
| **부작용** | 이 변경이 다른 부분에 영향을 주지 않는가? |

수정 사항이 더 이상 나오지 않을 때까지 반복합니다.

### 반복 리뷰 적용 시점 (5곳)

1. **Plan 후** — 구현 계획 반복 리뷰
2. **Build 각 phase 후** — 페이스별 코드 반복 리뷰 + 테스트
3. **모든 Build phase 후** — 전체 코드 반복 리뷰
4. **E2E Test 후** — 테스트 통과 후 전체 코드 반복 리뷰 (POST-TEST REVIEW)
5. **Sync 후** — 컨텍스트 미러 정확성 반복 검증 (SYNC VERIFY)

## E2E 테스트

**핵심: 실제 앱/웹 서버를 실행하여 테스트한다.** 단순 통합 흐름 체크가 아님.

### 실행 순서
1. `.env`에서 필요한 테스트 데이터(API 키, 토큰, URL) 확인
2. 필요한 환경변수가 없으면 **사용자에게 요청** (추측하지 않음)
3. 실제 개발 서버 / 앱 실행
4. **변경한 부분만 먼저 타겟 테스트** (빠른 피드백)
5. 타겟 테스트 통과 → **전체 E2E 스위트** 실행
6. 실패 시 → **진단**(출력 전문 읽기) → 근본 원인 수정 → 타겟 테스트 재실행

### 규칙
- 실제 웹 서버 또는 애플리케이션을 **반드시 실행** (mock/stub 금지)
- 테스트 데이터는 `.env` 기반 — 없으면 사용자에게 요청
- 런타임 패치 없이, 사용자 목적이 추가 개입 없이 달성되어야 함

### 실패 시 되돌림

**필수 첫 단계: DIAGNOSE** — 코드 수정 전에 반드시 테스트 출력을 읽고 진단

1. 테스트 출력 전문 읽기 (에러 메시지, 스택 트레이스, 실제 vs 기대값)
2. 테스트 코드의 assertion이 올바른지 확인
3. 근본 원인 식별: 구현/설계/조사 중 어디 문제인가?
4. progress file의 `test_findings`에 진단 결과 기록

그 후 라우팅:
- 구현 문제 → BUILD로 (무엇을 수정할지 구체적 진단과 함께)
- 설계 문제 → PLAN으로
- 조사 누락 → INVESTIGATE로

**금지:**
- 출력을 읽지 않고 BUILD로 돌아가기
- 실패 대응으로 assertion을 먼저 수정하기

## 컨텍스트 생존 (Anti-Compact)

긴 작업 중 대화 컨텍스트가 압축(compact)되면 초반 합의/교정이 사라집니다.

- 중요한 결정(조사 결과, plan, 교정 사항)은 **대화에 의존하지 말고 파일/Issue에 즉시 기록**
- compact 후에도 파일을 다시 읽으면 복구 가능

## 세션 종료 규칙

**모든 세션** 종료 시 (기능 사이클 완료 여부 무관):

목적: 다음 세션의 AI가 이번 세션의 학습을 물려받게 하는 것.

## PLAN 상세

### Scope Mode 선택

Plan 작성 전에 범위 모드를 선택한다. 조사 결과에 따라 원래 이슈 범위를 어떻게 다룰지 결정.

| 모드 | 언제 | 행동 | 리스크 |
|------|------|------|--------|
| **FULL_BUILD** | GREENFIELD/EXPANSION 이슈 — 포괄적 구현 필요 | 완전성 우선 — 계획된 모든 항목 구현, 필요시 확장 (사용자 승인) | 범위 확장 — 확장 경계를 명확히 정의 |
| **SELECTIVE_EXPANSION** | 대부분 잘 정의되었지만 1-2개 인접 영역 포함 필요 | 원래 범위 유지 + 명명된 항목 추가, 각각 근거 필수 | 미끄러운 경사 — 각 추가항목에 "빠지면 뭐가 깨지는지" 근거 필요 |
| **HOLD_SCOPE** | 범위가 잘 정의되고 조사에서 놀라운 점 없음 | 정확히 범위대로 구현 — 추가도 제거도 없음 | 터널 비전 — 범위 밖 항목을 미래 이슈로 명시 |
| **SCOPE_REDUCTION** | 조사 결과 이슈가 예상보다 단순하거나 일부 이미 해결됨 | 범위 축소 + 제거 항목과 이유 설명 | 조기 단순화 — 제거 항목마다 근거 인용 필수 |

### Phase 분할 원칙
- **각 Phase는 테스트 가능 단위로 쪼갤 것**
- 각 Phase에 검증 방법을 명시 (빌드 체크, 유닛 테스트, E2E, 수동 확인, 파일 리뷰 등)
- 로컬에서 테스트할 수 없는 Phase는 명시적으로 기재하고, 검증 가능한 범위를 표기
- 테스트 불가능한 작업과 가능한 작업을 같은 Phase에 묶지 않음

### Error & Rescue Map

각 Phase마다 아래 형식의 테이블을 작성한다:

| Method/Step | Failure Mode | Recovery Action | User Visible? |
|-------------|-------------|-----------------|:-------------:|

### 추가 규칙
- **대안 검토**: 검토한 대안과 왜 버렸는지 한 줄씩 명시 (ADR 형식)
- **사전 실패 시나리오 (pre-mortem)**: 이 계획이 실패하는 시나리오 3가지 → 각각 계획 조정 또는 수용된 리스크로 문서화
- **막연한 표현 금지**: 지표 없이 쓰지 않음 (`'빠르게'` → `'p99 < 200ms'`, `'개선'` → 측정 가능한 결과)
- **다이어그램 요구사항**:
  - **필수**: 3개 이상 컴포넌트 상호작용 시 아키텍처 다이어그램 + 데이터 흐름 다이어그램
  - **조건부**: 상태 머신, 에러 흐름, 배포 다이어그램 — 관련 있으면 포함
  - **생략**: 단일 파일 변경, 크로스 컴포넌트 상호작용 없음

## AI 자율 실행 구간

plan 승인 후 BUILD → REVIEW → E2E TEST → POST-TEST REVIEW → **SYNC (사용자 확인)** → SYNC VERIFY → REPORT → COMMIT을 실행합니다.
**SYNC에서는 사용자 확인 후** 진행합니다 (gate).
BUILD에서는 **Phase별 반복 리뷰 + 테스트 통과 후** 다음 Phase로 진행하고, **모든 Phase 완료 후 전체 반복 리뷰**를 수행합니다.

### SYNC 규칙
- 이슈 작업 중 시행착오를 컨텍스트에 반영
- **기존에 동일 내용이 있는지 먼저 확인** 후 작성 (중복/분산 방지)
- 업데이트 내용을 사용자에게 확인받고 진행
- **context 파일 변경 시 검증 테스트 필수**: workflows/*.yaml, skills/*/SKILL.md, context/*.json, hooks, .users/ 미러 변경 → 커밋 전에 검증 스크립트 실행 (YAML 파싱, 필수 섹션, .agents ↔ .users 일관성, 참조 파일 존재)

### 컨텍스트 라우팅 (어디에 쓸지)

| 저장소 | 언제 |
|-------|------|
| `agents-rules.json` | 이 프로젝트의 **모든** AI 세션에 강제할 규칙 — 아키텍처 결정, 코딩 컨벤션, 검증된 패턴 |
| `lessons-learned.yaml` | 실수/교정/반복 실수 — 세션 중 교정 받으면 즉시 기록 (anti-compact) |
| `CLAUDE.md / AGENTS.md` | 세션 진입점 규칙 — 모든 프로젝트(root) 또는 프로젝트 구조 정의 |
| `.env` 파일 | 시크릿, API 키, 환경변수 — 컨텍스트 파일에 절대 기록 금지 |
| **아무데도 쓰지 말 것** | 코드에 있는 것, git 로그에 있는 것, MEMORY.md (Claude Code 자동 관리) |

**판단 기준**: "이 프로젝트의 모든 AI 세션에 강제해야 하나?" → agents-rules. "내가 실수한 거라 반복하면 안 되나?" → lessons-learned. "모든 프로젝트에 필요한가?" → root CLAUDE.md.

## Git 연동

- 모든 사이클은 GitHub Issue에서 시작
- 브랜치 네이밍: `issue-{번호}-{설명}`

### 작업 공간 격리

| 모드 | 사용 시점 | 명령어 |
|------|----------|--------|
| **워크트리** (기본값) | 동시 작업 — 같은 프로젝트에서 여러 이슈를 병렬 진행 | `git worktree add ../{project}-issue-{N}-{desc} issue-{N}-{desc}` |
| **브랜치만** | 단독 작업 — 해당 레포에서 이슈 하나만 진행 | `git checkout -b issue-{N}-{desc}` |

- 조사 결과, 범위, plan을 Issue 코멘트로 기록
- **푸시 전 필수 검증** (git push 전에 반드시 실행):
  1. `git status` — 빌드에서 참조하는 파일(이미지, 폰트 등)이 모두 git 추적 중인지 확인
  2. Vercel 웹 프로젝트(aiedu, naia, about, admin 등): `pnpm build` / `npm run build` 통과
  3. Vercel 웹 프로젝트: 로컬 서버 기동 후 수정한 페이지 열고 브라우저 콘솔 에러 없는지 확인
  4. 위 3단계 통과 후에만 `git push`
  - **주의**: 로컬 빌드 통과 ≠ Vercel 빌드 통과 — 파일이 로컬에 있어도 git 미추적이면 Vercel에서 ENOENT
- 커밋 후 레포 유형에 따라 분기:
  - **nextain 레포** (Luke = 메인테이너): main에 직접 push — PR 생성 없음 (pr-guard 훅이 `gh pr create` 차단)
  - **외부/업스트림 기여**: PR 생성 → Issue에 링크
- 전체 의사결정 과정이 Issue 타임라인에 투명하게 남음

## CLOSE 상세

이슈 작업의 최종 마무리. 사용자 확인 없이 이슈를 닫을 수 없다 (게이트).

1. 이번 이슈 작업의 교훈(lessons-learned) 재검토
2. Phase-by-phase 완료보고서를 Issue 댓글로 작성:
   - 각 단계에서 무엇을 했는지
   - 통계 (패스 수, 수정 건수, 에이전트 수 등)
   - 주요 결정사항
   - 산출물 목록
3. 사용자에게 보고서 제시 + 확인 요청
4. 사용자 확인 후 Issue close
