## 2. 기술분야

본 발명은 인공지능 기반 소프트웨어 품질 보증 분야에 관한 것으로, 더 구체적으로는 복수의 독립적 AI 에이전트가 디지털 산출물(소스 코드, 설계 문서, 분석 보고서 등)을 병렬로 검증하는 멀티에이전트 상호 검증 시스템에서, 리뷰 에이전트의 출력 품질을 도메인 인지 방식으로 동적 평가하고, 품질 저하가 감지된 에이전트를 합의 기반으로 퇴출 및 교체하며, 교체 에이전트에 대한 편향 전파를 방지하는 방법 및 장치에 관한 것이다.

본 발명은 또한, 리뷰 에이전트가 리뷰 요청자 에이전트의 컨텍스트 드리프트(context drift)를 감지하여 검증 프로세스의 신뢰성을 유지하는 방법에 관한 것이다.

## 3. 발명의 배경이 되는 기술

### 3.1 기술적 배경

대규모 언어 모델(LLM)의 발전에 따라, 소프트웨어 개발 파이프라인에서 AI 에이전트를 활용한 자동화된 코드 리뷰, 설계 문서 검증, 보안 감사 등의 수요가 급증하고 있다. 단일 AI 에이전트에 의한 리뷰는 자기 검증 편향(self-review bias), 검증되지 않은 "클린" 선언, 장기 세션에서의 컨텍스트 드리프트, 확인되지 않은 주장(AI SLOP) 등의 구조적 약점을 가진다.

이를 해결하기 위해 복수의 AI 에이전트를 활용하는 멀티에이전트 시스템이 제안되었으나, 기존 접근법들은 다음과 같은 한계를 가진다.

### 3.2 선행기술 및 그 한계

#### 3.2.1 미국 특허출원 US20250259042A1 (SMASH — Smart Multi-Agent System Health Monitoring)

상기 선행기술은 멀티에이전트 시스템에서 에이전트의 건강 상태를 모니터링하는 방법을 개시하며, CPU 사용률, GPU 활용도, 메모리 소비, 응답 지연시간 등의 인프라 지표(infrastructure metrics)를 기반으로 에이전트의 건강도를 평가하고, 건강도가 저하된 에이전트를 새 인스턴스(fresh instance)로 교체(스폰)하는 기술을 포함한다.

그러나 상기 선행기술은 다음과 같은 한계를 가진다:
- **인프라 지표 의존**: 에이전트의 실제 출력 품질이 아닌 하드웨어/소프트웨어 인프라 상태만을 측정한다. LLM 기반 에이전트는 인프라 지표가 정상이더라도 환각(hallucination), 컨텍스트 소실, 도메인 이탈 등으로 인해 저품질 출력을 생성할 수 있으며, 이는 인프라 지표로 감지할 수 없다.
- **피어 합의 부재**: 에이전트 교체 결정이 중앙 모니터링 시스템에 의해 일방적으로 이루어지며, 동료 에이전트들의 합의(peer consensus)를 거치지 않는다.
- **도메인 인지 부재**: 에이전트의 전문 도메인(보안, 접근성, 성능 등)을 고려하지 않으므로, 전문 도메인 내의 정당한 발견사항이 다른 비전문 에이전트에 의해 기각되는 문제(도메인 전문가 오퇴출)를 방지할 수 없다.

#### 3.2.2 Fortytwo — 피어 랭킹 합의 시스템

Fortytwo는 분산 AI 추론 네트워크에서 피어 간 랭킹 합의 및 평판 가중 메커니즘을 활용하는 시스템이다. 각 에이전트가 다른 에이전트의 추론 결과를 평가하고, 누적 평판 점수에 따라 가중치를 부여하는 방식으로 합의에 도달한다.

그러나 상기 선행기술은 다음과 같은 한계를 가진다:
- **추론 합의 한정**: 범용 추론(reasoning) 작업에 대한 결과 합의에 초점을 맞추며, 소스 코드나 설계 문서 등 구조화된 산출물의 체계적 검증(verification)에는 적용되지 않는다.
- **스트라이크/도메인 면제 부재**: 기각된 발견사항에 대한 누적 벌점(스트라이크) 시스템이 없고, 에이전트의 전문 도메인을 고려한 면제 메커니즘이 없다.
- **교체 프로토콜 부재**: 저품질 에이전트를 동적으로 퇴출하고 새 에이전트로 교체하는 메커니즘이 없으며, 특히 교체 시 편향 전파를 방지하는 안티앵커링(anti-anchoring) 기법이 없다.

#### 3.2.3 Agent Drift (2026) — 행동 드리프트 정량화 시스템

Agent Drift는 12차원 에이전트 상태 지표(Agent State Indicators, ASI)를 사용하여 AI 에이전트의 시간 경과에 따른 행동 드리프트를 정량적으로 측정하는 시스템이다.

그러나 상기 선행기술은 다음과 같은 한계를 가진다:
- **측정 전용**: 드리프트를 측정할 뿐, 측정 결과를 기반으로 에이전트를 교체하거나 기각하는 액션 메커니즘이 없다.
- **자체 드리프트 한정**: 에이전트 자신의 행동 드리프트만을 측정하며, 리뷰어 에이전트가 요청자 에이전트의 컨텍스트 드리프트를 감지하는 방향(리뷰어→요청자)의 드리프트 검출은 지원하지 않는다.
- **검증 프로세스 통합 부재**: 멀티에이전트 상호 검증의 라운드 루프, 투표, 토론 등의 구조화된 파이프라인과 통합되지 않는다.

### 3.3 해결해야 할 기술적 과제

상기 선행기술들의 한계를 종합하면, 멀티에이전트 상호 검증 시스템에서 다음의 기술적 과제가 존재한다:

1. **출력 품질 기반 동적 에이전트 관리**: 인프라 지표가 아닌, 에이전트가 생성한 실제 출력(리뷰 보고서)의 구조적 특성으로부터 에이전트 품질을 추가 LLM 호출 없이(제로비용으로) 판별하는 방법
2. **도메인 인지 퇴출 메커니즘**: 전문 도메인 내의 정당한 발견사항과 도메인 이탈 발견사항을 구별하여, 전문가 에이전트의 부당한 퇴출을 방지하는 방법
3. **안티앵커링 교체**: 퇴출된 에이전트의 기각/논쟁 이력이 교체 에이전트에 편향으로 전파되는 것을 차단하는 방법
4. **양방향 드리프트 감지**: 리뷰 에이전트가 요청자 에이전트의 컨텍스트 드리프트를 감지하고 프로세스를 선택적으로 중단하는 방법
5. **이중소스 스트라이크 병합**: 자동기각(Phase 2)과 공식기각(Phase 3 토론 후)의 서로 다른 경로에서 발생하는 스트라이크를 단일 카운터로 병합하는 방법

## 4. 발명의 요약

### 4.1 발명의 목적

본 발명은 멀티에이전트 상호 검증 프레임워크에서, 리뷰 에이전트의 출력 품질을 도메인 인지 방식으로 평가하고, 품질 저하 에이전트를 스트라이크 기반 임계값으로 식별하고, 오케스트레이터 평가 및 사용자 확인을 거쳐 동적 퇴출 및 교체하되, 교체 에이전트에 대한 편향 전파를 방지하는 방법 및 장치를 제공하는 것을 목적으로 한다.

### 4.2 발명의 구성 개요

본 발명의 전체 파이프라인은 아래와 같다:

```
[리뷰 대상 산출물] ──→ [Phase 1: 독립 리뷰]
                        │
                        │ R개의 리뷰 에이전트가 병렬로 독립 분석
                        │ 각 에이전트는 격리된 컨텍스트에서 수행
                        ▼
                  [Phase 2: 투표]
                        │
                        ├─ 발견사항 매칭 (파일 경로 + 라인 범위 오버랩)
                        ├─ 분류: 확인(CONFIRMED) / 논쟁(CONTESTED) / 자동기각(AUTO-DISMISSED)
                        ├─ 자동기각 시 → 도메인 일관성 검사
                        │   ├─ 도메인 일치 → 스트라이크 면제
                        │   └─ 도메인 불일치 → 스트라이크 +1 (auto-dismiss 경로)
                        ├─ 건강 점수 산출 (제로비용 Tier 1 + 저비용 Tier 2)
                        │   └─ 7개 신호 가중합 → 합성 점수 (0-100)
                        └─ 컨텍스트 드리프트 검사 (REQUESTER_CONTEXT_DRIFT)
                        │
                        ▼
                  [Phase 3: 선택적 토론] (CONTESTED 발견사항만)
                        │
                        ├─ 교차 심문 에이전트 스폰 (상위 모델 사용)
                        ├─ 최대 3 서브라운드 토론
                        ├─ 최종 투표 → 확인 또는 기각
                        └─ 공식기각 시 → 제출자에게 스트라이크 +1 (formal-dismiss 경로)
                        │
                        ▼
                  [스트라이크 병합]
                        │
                        ├─ auto-dismiss 스트라이크 + formal-dismiss 스트라이크
                        └─ → 단일 strike_count 카운터에 병합
                        │
                        ▼
                  [퇴출 결정] (strike_count >= 임계값)
                        │
                        ├─ 오케스트레이터 평가 + 사용자 확인 후 퇴출 결정
                        └─ 퇴출 확정 시 → 안티앵커링 교체
                            ├─ 확인된 발견사항만 전달
                            └─ 기각/논쟁 이력 차단
                        │
                        ▼
                  [가드]
                        │
                        ├─ R=2 가드: 공식기각 비활성화 (무한루프 방지)
                        └─ 전원 플래그 가드: 모든 에이전트 저품질 시 사용자 개입
                        │
                        ▼
                  [라운드 결과]
                        │
                        ├─ 확인된 발견사항 0건 → clean_count 증가
                        │   └─ clean_count >= 임계값 → 리뷰 완료
                        └─ 확인된 발견사항 존재 → 사용자 조치 후 다음 라운드
```

### 4.3 발명의 효과 개요

본 발명에 의하면, (1) 추가 LLM 호출 없이 리뷰 에이전트의 출력 품질을 정량적으로 판별할 수 있고, (2) 도메인 전문가의 정당한 발견사항을 보호하면서 저품질 에이전트만을 선택적으로 퇴출할 수 있으며, (3) 교체 에이전트에 대한 편향 전파를 차단하여 독립적 검증을 유지할 수 있고, (4) 리뷰어가 요청자의 컨텍스트 드리프트를 감지하여 무효 컨텍스트 기반 검증을 조기에 차단할 수 있다.

## 5. 청구범위

### [청구항 1] (독립항 — 방법: 동적 퇴출)

복수의 독립 AI 리뷰 에이전트가 디지털 산출물을 병렬로 검증하는 멀티에이전트 상호 검증 프레임워크에서, 저품질 리뷰 에이전트를 도메인 인지 방식으로 동적 퇴출 및 교체하는 방법으로서, 프로세서 및 메모리를 포함하는 컴퓨터 시스템에 의해 수행되는 하기 단계를 포함하는 방법:

(a) **독립 리뷰 단계**: 상기 복수의 리뷰 에이전트 각각이, 서로 격리된 컨텍스트에서, 상기 디지털 산출물에 대한 발견사항(finding)을 독립적으로 생성하되, 각 발견사항은 적어도 파일 경로 및 라인 범위를 포함하는 위치 정보와 심각도 수준을 포함하는 구조화된 보고서로 출력하는 단계;

(b) **발견사항 매칭 및 투표 단계**: 상기 복수의 리뷰 에이전트로부터 수집된 발견사항들을, 코드 위치 오버랩(동일 파일 내 라인 범위가 1개 이상의 공유 라인을 가지는 경우)을 기준으로 매칭하고, 매칭된 발견사항 그룹을 리뷰 에이전트 수(R)에 기반한 투표로 확인(CONFIRMED), 논쟁(CONTESTED), 또는 기각(AUTO-DISMISSED)으로 분류하되, R이 3 이상인 경우 단 1명의 리뷰 에이전트만 제출하고 매칭이 없는 발견사항을 자동기각하고, R이 2인 경우 자동기각을 비활성화하여 논쟁으로 분류하는 단계;

(c) **도메인 일관성 검사 단계**: 상기 자동기각된 발견사항 각각에 대해, 해당 발견사항을 제출한 리뷰 에이전트의 프로필에 정의된 전문 도메인(expertise)과 상기 발견사항이 속하는 도메인을 비교하여, 도메인 일관성 여부를 판정하는 단계;

(d) **도메인 인지 스트라이크 관리 단계**: 상기 도메인 일관성 검사 결과, 도메인이 일치하는 경우 해당 리뷰 에이전트에 대한 스트라이크 누적을 면제하고, 도메인이 불일치하는 경우 해당 리뷰 에이전트의 스트라이크 카운터를 1 증가시키는 단계;

(e) **이중소스 병합 단계**: 상기 (d) 단계에서 자동기각 경로로 누적된 스트라이크와, 상기 논쟁 발견사항에 대한 선택적 토론 후 공식기각 투표에 의해 누적된 스트라이크를, 리뷰 에이전트별 단일 스트라이크 카운터에 병합하는 단계;

(f) **퇴출 결정 단계**: 상기 스트라이크 카운터가 미리 설정된 임계값(strikes_before_dismissal)에 도달한 리뷰 에이전트에 대해, 오케스트레이터가 기각된 발견사항과 실제 산출물을 대조하여 품질 저하 여부를 평가하고, 퇴출(DISMISS) 또는 유지(KEEP) 권고를 사용자에게 제시하여, 사용자 확인 후 퇴출 여부를 결정하는 단계;

(g) **안티앵커링 교체 단계**: 상기 퇴출이 결정된 리뷰 에이전트를 대체할 새로운 리뷰 에이전트를 생성(스폰)하되, 상기 새로운 리뷰 에이전트에게 확인(CONFIRMED)된 발견사항만을 전달하고, 기각된 발견사항 및 논쟁 이력은 차단하여, 선행 에이전트의 편향이 전파되는 것을 방지하는 단계;

(h) **R=2 가드 단계**: 리뷰 에이전트의 수가 2인 경우, 상기 공식기각 투표를 비활성화하여, 확인-기각 무한루프를 방지하는 단계; 및

(i) **전원 플래그 가드 단계**: 모든 리뷰 에이전트가 저품질 플래그 상태(자동기각된 발견사항만 존재하고 확인된 발견사항이 없는 상태)에 해당하는 경우, 자동 처리를 중단하고 사용자 개입을 강제하는 단계

를 포함하는, 멀티에이전트 상호 검증 프레임워크에서의 도메인 인지 기반 동적 참가자 관리 방법.

### [청구항 2] (독립항 — 방법: 컨텍스트 드리프트 감지)

멀티에이전트 상호 검증 시스템에서, 리뷰 에이전트가 요청자 에이전트의 컨텍스트 드리프트를 감지하여 검증 프로세스의 신뢰성을 유지하는 방법으로서, 프로세서 및 메모리를 포함하는 컴퓨터 시스템에 의해 수행되는 하기 단계를 포함하는 방법:

(a) **검증 단계**: 복수의 리뷰 에이전트 각각이, 리뷰 요청자 에이전트가 제공한 분석 참조 정보(파일 경로, 코드 라인 번호, 기술적 주장)와, 상기 분석 참조 정보가 지시하는 실제 디지털 산출물(소스 코드, 설계 문서)의 현재 상태를 대조하는 단계;

(b) **드리프트 보고 단계**: 상기 대조 결과, 요청자 에이전트의 분석 참조 정보와 실제 산출물 사이에 불일치가 발견된 경우, 해당 리뷰 에이전트가 REQUESTER_CONTEXT_DRIFT 판정을 발행하되, 불일치 증거(요청자가 참조한 내용과 실제 파일 내용의 차이)를 포함하는 구조화된 보고서를 출력하는 단계;

(c) **선택적 일시정지 단계**: 1명 이상의 리뷰 에이전트가 상기 REQUESTER_CONTEXT_DRIFT를 보고한 경우, 진행 중인 검증 프로세스를 선택적으로 일시정지하고, 사용자에게 드리프트 감지 사실 및 증거를 통지하는 단계; 및

(d) **복구 단계**: 상기 요청자 에이전트에게 관련 산출물의 재읽기를 권장하고, 사용자의 확인 후 검증 프로세스를 재개하되, 재개 시 드리프트가 감지된 라운드의 다음 라운드부터 실행하는 단계

를 포함하는, 멀티에이전트 상호 검증 시스템에서의 컨텍스트 드리프트 감지 방법.

### [청구항 3] (독립항 — 장치: 동적 퇴출 시스템)

디지털 산출물을 검증하는 멀티에이전트 상호 검증 장치로서, 다음을 포함하는 장치:

프로세서; 상기 프로세서에 연결된 메모리; 및 상기 메모리에 저장되어 상기 프로세서에 의해 실행되는 명령어를 포함하되,

상기 명령어는 상기 프로세서로 하여금:

(a) 복수의 독립 AI 리뷰 에이전트를 격리된 컨텍스트로 각각 스폰하여, 대상 디지털 산출물에 대한 발견사항을 병렬로 생성하게 하는 **독립 리뷰 모듈**;

(b) 상기 발견사항들을 코드 위치 오버랩 기반으로 매칭하고, 리뷰 에이전트 간 투표로 확인/논쟁/기각으로 분류하는 **합의 투표 모듈**;

(c) 기각된 발견사항에 대해, 제출 에이전트의 전문 도메인과 발견사항의 도메인 일치 여부를 검사하여, 일치 시 스트라이크 면제를 결정하는 **도메인 일관성 검사 모듈**;

(d) 자동기각 경로 및 공식기각 경로에서 발생한 스트라이크를 리뷰 에이전트별 단일 카운터에 병합하는 **이중소스 스트라이크 병합 모듈**;

(e) 스트라이크 카운터가 임계값에 도달한 리뷰 에이전트에 대해, 기각된 발견사항과 실제 산출물을 대조하여 품질 저하 여부를 평가하고, 사용자 확인 후 퇴출 여부를 결정하는 **퇴출 결정 모듈**;

(f) 퇴출된 에이전트를 대체하는 새 에이전트에게, 확인된 발견사항만 전달하고 기각/논쟁 이력을 차단하는 **안티앵커링 교체 모듈**;

(g) 리뷰 에이전트 수가 2인 경우 공식기각을 비활성화하고, 전 에이전트가 저품질 플래그 상태일 때 사용자 개입을 강제하는 **가드 모듈**;

(h) 각 리뷰 에이전트의 구조화된 보고서 메타데이터로부터 복수의 구조적 신호 및 텍스트 통계 신호를 추출하고, 이원적 정규화 후 가중합으로 합성 건강 점수를 산출하는 **건강 점수 엔진 모듈**; 및

(i) 리뷰 에이전트가 요청자 에이전트의 분석 참조 정보와 실제 산출물의 현재 상태를 대조하여 불일치를 감지하고, REQUESTER_CONTEXT_DRIFT 판정을 발행하며, 검증 프로세스를 선택적으로 일시정지하는 **컨텍스트 드리프트 감지 모듈**

을 수행하게 하는 것을 특징으로 하는, 멀티에이전트 상호 검증 장치.

### [청구항 4] (종속항 — 제로비용 품질 판별)

제1항의 방법에 있어서, 상기 리뷰 에이전트의 출력 품질을 추가 LLM(대규모 언어 모델) 호출 없이 판별하는 단계를 더 포함하되, 상기 품질 판별 단계는:

(a) **Tier 1 제로비용 신호 추출 단계**: 각 리뷰 에이전트의 구조화된 보고서 메타데이터로부터 다음의 4개 구조적 신호를 추출하는 단계:
- **Claim-to-Read 비율**: 발견사항에서 참조한 고유 파일 수를 보고서의 파일 읽기 목록(files_read) 섹션에 기재된 고유 파일 수로 나눈 값으로서, 임계값(기본값: 3.0)을 초과하는 경우 저품질 신호로 판정;
- **Specificity Score**: 파일 경로 및 라인 번호를 명시적으로 기재한 발견사항의 수를 전체 발견사항 수로 나눈 비율로서, 임계값(기본값: 0.30) 미만인 경우 저품질 신호로 판정;
- **Verifiability Score**: 발견사항이 인용한 코드 참조(파일:라인)에 대해 해당 파일의 해당 라인을 실제로 읽어 인용 내용이 존재하는지 검증한 비율로서, 임계값(기본값: 0.70) 미만인 경우 저품질 신호로 판정;
- **Prompt Echo Ratio**: 발견사항 중 리뷰 요청의 원문 기술(review target description)과 동일하거나 실질적으로 동일한 내용을 반복하는 발견사항의 비율로서, 임계값(기본값: 0.50)을 초과하는 경우 저품질 신호로 판정;

(b) **Tier 2 저비용 신호 추출 단계**: 보고서 텍스트로부터 다음의 3개 텍스트 통계 신호를 추출하는 단계:
- **Cross-Agent Agreement**: 해당 리뷰 에이전트의 발견사항 중 타 리뷰 에이전트와 매칭되는 발견사항의 비율로서, 임계값(기본값: 0.30) 미만이 2라운드 이상 연속되는 경우 저품질 신호로 판정;
- **Novelty Ratio**: 이전 라운드에 존재하지 않았던 새로운 발견사항의 비율로서, 임계값(기본값: 0.50) 미만인 경우 저품질 신호로 판정;
- **Hedge Density**: 회피적 표현("it's worth noting", "arguably", "consider", "might", "potentially", "perhaps" 등)의 100단어당 출현 빈도로서, 임계값(기본값: 3.0) 초과 시 저품질 신호로 판정;

(c) **이원적 정규화 단계**: 상기 7개 신호를 0.0 내지 1.0 범위로 정규화하되, 비율 신호(Claim-to-Read, Prompt Echo, Hedge Density)에 대해서는 `1.0 - min(값/임계값, 1.0)` 수식으로 정규화하고, 비례 신호(Specificity, Verifiability, Cross-Agent Agreement, Novelty)에 대해서는 원래 값을 그대로 사용하는 단계;

(d) **합성 건강 점수 산출 단계**: 상기 정규화된 7개 신호에 각각 미리 설정된 가중치(Claim-to-Read: 0.15, Specificity: 0.15, Verifiability: 0.25, Prompt Echo: 0.10, Hedge Density: 0.10, Cross-Agent Agreement: 0.15, Novelty: 0.10)를 적용하여 가중합을 구하고, 100을 곱하여 0 내지 100 범위의 합성 건강 점수를 산출하는 단계; 및

(e) **라운드 1 초기화 단계**: 첫 번째 라운드에서 Cross-Agent Agreement 및 Novelty Ratio 값을 1.0으로 초기화하여, 비교 기준선 부재에 따른 오탐(false positive)을 방지하는 단계

를 포함하는 것을 특징으로 하는 방법.

### [청구항 5] (종속항 — 구조적 제약 페르소나 기반 검증)

제4항의 방법에 있어서, 상기 품질 판별 시스템의 탐지 정확도를 검증하는 단계를 더 포함하되, 상기 검증 단계는:

(a) **구조적 제약 페르소나 생성 단계**: AI 에이전트의 도움편향(helpfulness bias)을 우회하기 위해, "저품질 리뷰를 생성하라"는 직접적 지시 대신, 구조적 제약을 부여하여 간접적으로 저품질 패턴을 유도하는 3종의 테스트 페르소나를 정의하는 단계로서:
- **Domain Amnesiac 페르소나**: 리뷰 대상의 실제 도메인과 무관한 전문성을 부여(예: Rust 시스템 코드에 대해 WCAG 웹 접근성 전문가 역할을 부여)하여, 도메인 불일치 발견사항을 자연스럽게 유도하는 페르소나;
- **Scope Constrained 페르소나**: 분석 범위를 극단적으로 제한(예: 파일의 1-20번 라인만 분석하도록 지시)하여, 낮은 Specificity Score 및 좁은 커버리지의 발견사항을 유도하는 페르소나;
- **Overcautious 페르소나**: 과도한 위협 모델을 부여(예: 국가 수준의 공격자를 가정한 위협 분석을 지시)하여, 모든 발견사항을 CRITICAL/HIGH로 분류하는 과잉 경고 패턴을 유도하는 페르소나;

(b) **탐지율 검증 단계**: 상기 각 페르소나를 리뷰 에이전트로 투입하여 실제 리뷰를 수행하게 한 후, 상기 제4항의 품질 신호 및 합성 건강 점수가 각 페르소나의 저품질 패턴을 정확히 탐지하는지 검증하되, 각 페르소나가 트리거하는 품질 신호를 매트릭스로 매핑하여 탐지율을 산출하는 단계

를 포함하는 것을 특징으로 하는 방법.

### [청구항 6] (종속항 — 선택적 토론)

제1항의 방법에 있어서, 상기 (b) 단계에서 논쟁(CONTESTED)으로 분류된 발견사항에 대해:

(a) 논쟁 발견사항과 모든 리뷰 에이전트의 입장을 입력으로, 상기 리뷰 에이전트보다 상위 능력 계층의 모델(arbitration tier)을 사용하는 교차 심문 에이전트를 스폰하여 실제 소스 코드를 읽고 증거를 수집하게 하는 단계;

(b) 최대 미리 설정된 서브라운드 수(기본값: 3)까지 토론을 진행하되, 토론 후 원래 리뷰 에이전트들의 최종 투표를 수집하여, 과반수 기준(ceil(R/2))으로 확인 또는 기각을 결정하는 단계; 및

(c) 짝수 R에서 동수(R/2 대 R/2)인 경우 확인(CONFIRMED)으로 결정하여, 거짓 음성(false negative, 놓친 버그)보다 거짓 양성(false positive, 과잉 보고)을 선호하는 방향으로 편향시키는 단계

를 더 포함하는 것을 특징으로 하는 방법.

### [청구항 7] (종속항 — 심각도 기반 투표 임계값 차등)

제1항의 방법에 있어서, 상기 (b) 단계의 투표에서 발견사항의 심각도 수준에 따라 확인 임계값을 차등 적용하되:

CRITICAL 심각도의 발견사항에 대해서는 합의 전략(consensus strategy) 설정과 무관하게 전원합의(ALL R)를 요구하고, HIGH/MEDIUM/LOW 심각도의 발견사항에 대해서는 프로필에 설정된 합의 전략(전원합의 또는 과반수)을 따르는 것을 특징으로 하는 방법.

### [청구항 8] (종속항 — 모델 계층화)

제1항의 방법에 있어서, 상기 멀티에이전트 상호 검증 프레임워크의 각 단계에서 사용되는 AI 모델의 능력 계층을 차등 적용하되:

Phase 1(독립 리뷰)에서는 판단(judgment) 계층의 모델을 사용하고, Phase 3(선택적 토론)에서는 중재(arbitration) 계층의 상위 모델을 사용하며, 생성(요약, 보고서 초안) 작업에서는 생성(generation) 계층의 하위 모델을 사용하여, 비용 효율성과 판단 정확도를 동시에 최적화하는 것을 특징으로 하는 방법.

### [청구항 9] (종속항 — 프로필 기반 구성)

제1항의 방법에 있어서, 상기 멀티에이전트 상호 검증 프레임워크의 모든 가변 파라미터가 YAML 형식의 프로필 파일로 구성 가능하되, 상기 프로필은:

리뷰 에이전트의 수, 각 에이전트의 전문 도메인 및 리뷰 전략, 합의 전략(전원합의/과반수/초과반수/임계값), 연속 클린 라운드 수, 최대 토론 서브라운드 수, 퇴출 전 스트라이크 임계값, 교체 정책, 품질 신호 임계값, 증거 요구사항, 모델 정책, 및 비용 예산을 포함하며,

프로필 간 상속(extends) 메커니즘을 지원하여 기반 프로필의 파라미터를 자식 프로필이 선택적으로 오버라이드할 수 있는 것을 특징으로 하는 방법.

### [청구항 10] (종속항 — 이벤트 로깅)

제1항의 방법에 있어서, 상기 방법의 각 단계에서 발생하는 이벤트를 JSONL(JSON Lines) 형식의 추가 전용(append-only) 이벤트 로그에 기록하되, 상기 이벤트 로그는 적어도:

리뷰 시작(REVIEW_STARTED), 리뷰어 스폰(REVIEWER_SPAWNED), 보고서 제출(REPORT_SUBMITTED), 발견사항 확인(FINDING_CONFIRMED), 발견사항 기각(FINDING_DISMISSED), 발견사항 논쟁(FINDING_CONTESTED), 투표(VOTE_CAST), 건강 점수(HEALTH_SCORE), 에이전트 건강 플래그(AGENT_HEALTH_FLAGGED), 에이전트 퇴출(AGENT_DISMISSED), 컨텍스트 드리프트 감지(CONTEXT_DRIFT_DETECTED), 라운드 완료(ROUND_COMPLETED), 및 리뷰 완료(REVIEW_COMPLETED) 이벤트 유형을 포함하여, 사후 감사(post-hoc audit) 및 투표 오류 분석을 지원하는 것을 특징으로 하는 방법.

## 6. 발명의 상세한 설명

### 6.1 용어의 정의

본 명세서에서 사용되는 주요 용어는 다음과 같이 정의한다:

- **디지털 산출물(digital artifact)**: 소스 코드, 설계 문서, 분석 보고서, 기술 명세서 등 AI 에이전트가 생성하거나 검증하는 디지털 문서의 총칭
- **제로비용(zero-cost)**: 본 명세서에서 "제로비용"이란 추가 대규모 언어 모델(LLM) 추론 호출이 필요하지 않음을 의미한다. 파일 I/O 등 일반적인 연산 비용은 발생할 수 있으나, 에이전트 품질 판별을 위해 별도의 LLM API 호출(예: 평가 에이전트 스폰, 임베딩 모델 호출)을 요구하지 않는다는 것을 지칭한다
- **리뷰 에이전트(review agent)**: 디지털 산출물을 독립적으로 분석하여 발견사항을 생성하는 AI 에이전트. 각 리뷰 에이전트는 고유한 전문 도메인(expertise)과 리뷰 전략(strategy)을 가짐
- **요청자 에이전트(requester agent)**: 리뷰를 요청하는 AI 에이전트. 리뷰 대상 산출물의 원저자이거나, 원저자를 대리하는 에이전트
- **오케스트레이터(orchestrator)**: 멀티에이전트 상호 검증 프로세스 전체를 관리하는 중앙 조정 에이전트. 프로필 로딩, 라운드 관리, 투표 집계, 퇴출 결정 등을 수행
- **발견사항(finding)**: 리뷰 에이전트가 산출물에서 식별한 문제, 결함, 또는 개선사항. 위치 정보(파일 경로, 라인 범위), 심각도(CRITICAL, HIGH, MEDIUM, LOW), 설명, 증거를 포함
- **라운드(round)**: Phase 1(독립 리뷰), Phase 2(투표), Phase 3(선택적 토론)으로 구성된 검증 반복의 한 주기
- **스트라이크(strike)**: 리뷰 에이전트의 발견사항이 기각될 때 누적되는 벌점. 자동기각 경로와 공식기각 경로의 두 소스에서 발생
- **클린 라운드(clean round)**: 확인된 발견사항이 0건인 라운드
- **도메인 일관성(domain consistency)**: 리뷰 에이전트의 발견사항이 해당 에이전트의 전문 도메인 내에 있는지 여부
- **안티앵커링(anti-anchoring)**: 퇴출된 에이전트의 기각/논쟁 이력이 교체 에이전트에 전파되어 편향을 생성하는 것을 방지하는 기법
- **컨텍스트 드리프트(context drift)**: 에이전트가 참조하는 산출물의 상태가 실제 현재 상태와 불일치하는 현상

#### 6.2.1 프로필 로더

프로필 로더는 YAML 형식의 프로필 파일을 읽어 파싱하고 유효성을 검증한다. 프로필은 기반 프로필(`_base`)로부터 상속이 가능하며, 자식 프로필은 부모의 파라미터를 선택적으로 오버라이드한다. 구현 실시예에서, 기반 프로필은 다음의 기본값을 정의한다(참조: `.agents/profiles/_base.yaml`):

- 리뷰 에이전트 수: 3
- 합의 전략: unanimous (전원합의)
- 연속 클린 라운드: 2
- 최대 토론 서브라운드: 3
- 퇴출 전 스트라이크 임계값: 2
- 교체 정책: spawn_fresh (새 인스턴스 생성)
- 모델 정책: generation=haiku, judgment=sonnet, arbitration=opus

#### 6.2.2 라운드 관리자

라운드 관리자는 Phase 1 → Phase 2 → Phase 3 → Round Result의 순서로 각 라운드를 진행하며, 라운드 간 상태(클린 카운트, 스트라이크 카운터, 확인된 발견사항 목록 등)를 유지한다. 최대 라운드 수(기본값: 10)에 도달하면 리뷰를 종료한다.

### 6.3 상세 동작 — 독립항 A (동적 퇴출 방법)

#### 6.3.1 Phase 1: 독립 리뷰 단계 (청구항 1(a))

오케스트레이터는 프로필에 정의된 수(R)만큼의 리뷰 에이전트를 생성(스폰)한다. 각 리뷰 에이전트는 서로 격리된 컨텍스트에서 실행되며, 다른 에이전트의 존재나 발견사항을 인지하지 못한다.

구현 실시예에서, 각 리뷰 에이전트는 Agent 도구를 통해 독립적인 서브에이전트로 스폰되며, 시스템 프롬프트에 전문 도메인과 리뷰 전략이 명시된다. 예를 들어, `code-review` 프로필에서는 보안 전문가(security-expert), 정확성 분석가(correctness), SLOP 탐지기(slop-detector)의 3개 에이전트가 각각 다른 리뷰 전략으로 스폰된다.

SLOP 탐지기의 시스템 프롬프트(참조: `.agents/prompts/slop-detector.md`)는 "CLAIM VERIFICATION" 전략을 사용하며, 모든 주장에 대해 (1) 파일 참조 확인, (2) 존재 검증(실제 파일/라인 읽기), (3) 신규성 확인, (4) 회피적 표현 탐지, (5) 자기 모순 탐지, (6) 구체성 확인의 6단계 검증을 수행한다.

각 리뷰 에이전트는 구조화된 보고서를 출력하며, 보고서는 최소한 다음을 포함한다:
- 범위(Scope): 리뷰한 대상의 기술
- 파일 읽기 목록(Files read): 실제로 읽은 파일과 라인 범위
- 확인 항목(Checked): 점검한 항목과 결과
- 발견사항(Findings): 발견된 문제 목록 (심각도, 설명, 위치, 증거)
- 판정(Verdict): CLEAN, FOUND_ISSUES, 또는 REQUESTER_CONTEXT_DRIFT

#### 6.3.2 Phase 2: 투표 단계 (청구항 1(b))

##### 6.3.2.1 발견사항 매칭

오케스트레이터는 모든 리뷰 에이전트의 보고서를 수집한 후, 발견사항들을 코드 위치 오버랩 기반으로 매칭한다. 매칭 알고리즘은 다음과 같다:

1. 모든 서로 다른 리뷰 에이전트의 발견사항 쌍에 대해, 동일 파일을 참조하고 라인 범위가 1개 이상의 공유 라인을 가지면 MATCH로 판정
2. 복수 파일을 참조하는 발견사항의 경우, 임의의 공통 파일에서 라인 오버랩이 있으면 매칭
3. 모든 잠재적 매칭을 오버랩 점수 내림차순으로 정렬하여 탐욕적(greedy) 최선 우선 매칭 수행
4. 리뷰 에이전트 쌍당 1대1 매칭 (한 발견사항이 동일 리뷰 에이전트의 두 발견사항과 다중 매칭되지 않음)
5. 파일:라인 참조가 없는 발견사항은 의미적 유사성(동일 개념, 동일 코드 영역, 동일 우려 참조 여부)으로 매칭

##### 6.3.2.2 분류

매칭된 각 발견사항 그룹에 대해, R(해당 라운드에 보고서를 제출한 리뷰 에이전트 수)를 기준으로 분류한다:

| 조건 | 분류 | 비고 |
|------|------|------|
| 지지자 >= 확인 임계값 | CONFIRMED | |
| R >= 3 이고 1명만 제출, 매칭 없음 | AUTO-DISMISSED | 스트라이크 없음 (R >= 3에서만) |
| R = 2 이고 1명만 제출 | CONTESTED | 자동기각 비활성화 |
| 지지자 < 확인 임계값 | CONTESTED | Phase 3 토론으로 이동 |

심각도 기반 임계값 차등(청구항 7): CRITICAL 심각도의 발견사항은 합의 전략 설정과 무관하게 항상 전원합의(ALL R)를 요구한다. 이는 데이터 손실, 보안, 부정확한 출력 등 CRITICAL 문제의 거짓 음성 비용이 매우 높기 때문이다.

#### 6.3.3 도메인 일관성 검사 단계 (청구항 1(c))

자동기각된 각 발견사항에 대해, 오케스트레이터는 해당 발견사항을 제출한 리뷰 에이전트의 프로필에 정의된 `expertise` 필드와 발견사항이 속하는 도메인을 비교한다.

예를 들어, 보안 전문가(security-expert) 에이전트가 제출한 보안 관련 발견사항은, 다른 비보안 전문가 에이전트들이 해당 발견사항을 인지하지 못하여 자동기각되더라도, 도메인 일관성이 인정되어 스트라이크가 면제된다.

구현 실시예(참조: `.claude/skills/cross-review/SKILL.md`, Appendix 8A)에서, 도메인 불일치의 예시는 다음과 같다:
- 보안 리뷰 에이전트가 접근성(accessibility) 문제를 보고하는 경우
- 정확성 분석 에이전트가 국가 수준 위협(nation-state threat)을 보고하는 경우

#### 6.3.4 도메인 인지 스트라이크 관리 단계 (청구항 1(d))

오케스트레이터는 리뷰 에이전트별 `strike_count` 맵을 유지한다. 스트라이크 증가 규칙:
- 도메인 일관성 검사 결과 일치: 스트라이크 증가 없음. 로그에 `"strike_incremented": false, "reason": "domain-consistent solo finding"` 기록
- 도메인 일관성 검사 결과 불일치: `strike_count[reviewer_id]` += 1. 로그에 `"strike_incremented": true` 기록

확인된(CONFIRMED) 발견사항은 무죄 증거(exculpatory)로서, 스트라이크를 증가시키지 않는다.

건강 검사는 라운드의 모든 발견사항이 처리된 후 1회 실행되며, 개별 발견사항 처리 중에는 실행되지 않는다.

#### 6.3.5 Phase 3: 선택적 토론 및 공식기각 (청구항 1(e)의 공식기각 경로)

CONTESTED로 분류된 발견사항에 대해서만 토론이 진행된다(청구항 6). 토론 과정에서, Phase 3 투표 후 확인된 발견사항에 대해 리뷰 에이전트가 이의를 제기하면 공식기각 절차가 개시된다(참조: SKILL.md Appendix 8E):

1. 이의 제기 리뷰 에이전트가 공식기각을 신청
2. R=2인 경우: 공식기각 비활성화 (청구항 1(h)). R=2에서는 비제출자가 유일한 투표자이므로 항상 일방적 기각이 가능하여 확인→기각 무한루프가 발생
3. R >= 3인 경우: 발견사항 제출자를 제외한 모든 참가자가 투표
4. 기각이 확정되면 제출자에게 스트라이크 +1

##### 6.3.5.1 동수 처리 규칙 (청구항 6(c))

Phase 3 토론 후 최종 투표에서, 과반수 기준(ceil(R/2))을 적용할 때 짝수 R에서 동수(R/2 대 R/2)가 발생하는 경우, 본 발명은 확인(CONFIRMED)으로 결정한다. 이 설계 결정의 기술적 근거: 소프트웨어 검증에서 거짓 음성(실제 결함을 놓침)의 비용이 거짓 양성(결함이 아닌 것을 결함으로 보고)의 비용보다 크므로, 동수 시 발견사항을 표면화하는 방향으로 편향시킨다. 수학적으로, ceil(R/2)를 임계값으로 사용하면 R=4일 때 ceil(4/2)=2이므로 2명 지지로 확인되어 동수(2:2) 상황이 자동으로 CONFIRMED로 결정되며, 이는 별도의 동수 처리 규칙이 아닌 ceil 함수의 자연적 결과이다.

#### 6.3.6 이중소스 병합 단계 (청구항 1(e))

자동기각 경로(Phase 2, 6.3.4절)에서 누적된 스트라이크와 공식기각 경로(Phase 3, 6.3.5절)에서 누적된 스트라이크는 동일한 `strike_count` 맵에 병합된다. 따라서, 자동기각 스트라이크 1회 + 공식기각 스트라이크 1회 = 2회 = 기본 임계값에 도달한다.

이 병합 설계의 기술적 효과: 서로 다른 경로로 드러나는 품질 저하 신호를 통합적으로 추적하여, 한 경로에서만 임계값 미만인 에이전트도 다른 경로와 합산하여 적시에 퇴출할 수 있다.

#### 6.3.7 퇴출 결정 단계 (청구항 1(f))

스트라이크 카운터가 임계값(`strikes_before_dismissal`, 기본값: 2)에 도달하면:

1. 오케스트레이터가 AGENT_HEALTH_FLAGGED 이벤트를 로깅하고, 기각된 발견사항 목록을 포함
2. 오케스트레이터가 기각된 발견사항과 실제 소스 코드를 대조하여 발견사항이 실제로 근거 없는지(off-topic, ungrounded, hallucinated) 평가
3. DISMISS 또는 KEEP 권고를 사용자에게 제시
4. 사용자 확인 후 퇴출 또는 유지 결정

구현 실시예에서는 리뷰 에이전트를 재스폰하지 않고, 오케스트레이터가 모든 발견사항을 컨텍스트에 보유하고 있으므로 직접 품질을 평가한다.

#### 6.3.8 안티앵커링 교체 단계 (청구항 1(g))

퇴출이 결정된 리뷰 에이전트를 대체하는 새 에이전트(A')의 스폰 시:

- **전달하는 정보**: 리뷰 대상 산출물 + 확인된(CONFIRMED) 발견사항 목록
- **차단하는 정보**: 기각된 발견사항, 논쟁 이력, 선행 에이전트의 보고서
- A'는 동일한 프로필 리뷰 에이전트 스펙(expertise, prompt_ref, strategy)을 사용하되 완전히 새로운 컨텍스트에서 시작
- A'는 Round N+1부터 참여 (현재 라운드가 아님)
- 선행 에이전트 A의 보고서는 Round N+1의 매칭 풀에서 제외

안티앵커링의 기술적 근거: 기각된 발견사항을 전달하면, 교체 에이전트가 선행 에이전트의 기각 패턴을 부정적 앵커로 사용하여, 유사한 영역의 정당한 발견사항도 회피하는 편향(negative anchoring)이 발생한다. 반면, 확인된 발견사항은 전달하여 이미 알려진 문제를 재발견하는 중복 작업을 방지한다.

#### 6.3.9 R=2 가드 단계 (청구항 1(h))

리뷰 에이전트가 2명인 경우의 특수 처리:

- **Phase 2**: 자동기각(AUTO-DISMISSED) 비활성화. 1명만 제출한 발견사항은 CONTESTED로 분류 (6.3.2.2절 참조)
- **Phase 3**: 공식기각 비활성화. 비제출자가 유일한 투표자이므로 항상 일방적 기각이 가능하여, 확인→기각 무한루프 발생. Phase 3 토론만 진행하고 사후 기각 불허
- **Phase 3 과반수**: ceil(2/2) = 1이므로 발견사항 제출자의 투표만으로 확인 가능. 이는 R=2 설계상 의도된 동작

#### 6.3.10 전원 플래그 가드 단계 (청구항 1(i))

임의의 라운드에서 모든 리뷰 에이전트가 다음 조건을 모두 만족하는 경우:
- 1건 이상의 자동기각된 독립(solo) 발견사항이 존재하고
- 확인된 발견사항이 0건

오케스트레이터는 ALL_REVIEWERS_FLAGGED 이벤트를 로깅하고, 사용자에게 경고를 표시하며, 사용자 확인 없이는 다음 라운드로 진행하지 않는다.

이 가드의 기술적 목적: 모든 에이전트가 동시에 품질 저하된 경우(예: 리뷰 대상이 AI의 공통 사각지대에 해당하는 경우), 자동 프로세스가 거짓 CLEAN 결과를 생성하는 것을 방지한다.

### 6.4 상세 동작 — 독립항 B (컨텍스트 드리프트 감지)

#### 6.4.1 검증 단계 (청구항 2(a))

각 리뷰 에이전트는 Phase 1 독립 리뷰 중에, 요청자 에이전트가 제공한 분석 참조 정보를 산출물의 현재 상태와 대조한다. 구체적으로:

- 요청자가 "browser.rs:61에 libc::kill이 cfg 가드 밖에 있다"고 기술한 경우, 리뷰 에이전트는 실제로 browser.rs:61을 읽어 해당 코드가 현재도 cfg 가드 밖에 있는지 확인
- 요청자가 참조한 함수가 이름 변경(renamed)되었거나 삭제(deleted)된 경우 불일치 감지

이 감지는 리뷰 에이전트의 정상적 리뷰 과정에서 자연스럽게 수행되며, 추가 비용이 발생하지 않는다(리뷰 에이전트는 이미 소스 코드를 읽어야 하므로).

#### 6.4.2 드리프트 보고 단계 (청구항 2(b))

불일치가 감지되면, 리뷰 에이전트는 보고서의 판정(Verdict)을 REQUESTER_CONTEXT_DRIFT로 설정하고, 불일치 증거를 포함한다. 증거에는 요청자가 참조한 내용과 실제 파일 내용의 구체적 차이가 기술된다.

#### 6.4.3 선택적 일시정지 단계 (청구항 2(c))

오케스트레이터는 Round Result 단계에서 드리프트 보고를 검사한다:
- 1명 이상의 리뷰 에이전트가 REQUESTER_CONTEXT_DRIFT를 보고한 경우, CONTEXT_DRIFT_DETECTED 이벤트를 로깅
- 사용자에게 드리프트 감지 사실, 감지한 리뷰 에이전트 ID, 증거를 표시
- 사용자에게 현재 컨텍스트로 계속할지(yes) 또는 중단하여 갱신할지(no)를 선택하게 함

#### 6.4.4 복구 단계 (청구항 2(d))

사용자가 중단을 선택한 경우:
- 요청자 에이전트에게 관련 산출물을 재읽기하도록 권장
- 사용자가 갱신 완료를 확인한 후, Round N+1부터 검증 프로세스를 재개

이 설계의 기술적 특징: 기존 기술(Agent Drift)이 에이전트 자신의 드리프트만 측정하는 것과 달리, 본 발명은 리뷰어→요청자 방향의 교차 드리프트 감지를 구현한다. 이는 멀티에이전트 시스템의 고유한 이점으로서, 단일 에이전트 시스템에서는 달성할 수 없다.

### 6.5 상세 동작 — 종속항 C (제로비용 품질 판별)

#### 6.5.1 Tier 1 제로비용 신호 (청구항 4(a))

4개의 구조적 신호는 보고서의 메타데이터(파일 읽기 목록, 발견사항의 파일:라인 참조 등)에서 추출되며, 추가 LLM 호출이나 외부 API 호출 없이 순수하게 구조적 분석만으로 계산된다:

**Claim-to-Read 비율**:
```
ratio = unique_files_referenced_in_findings / unique_files_in_files_read_section
```
분모가 0인 경우(파일을 읽지 않은 경우) 비율은 무한대(infinity)로 설정되어 자동 플래그.

**Specificity Score**:
```
score = findings_with_file_line / total_findings
```
파일:라인 위치가 명시되지 않은 발견사항은 구체성이 부족한 것으로 평가.

**Verifiability Score**:
```
score = verified_citations / total_citations
```
각 발견사항이 인용한 파일:라인을 실제로 읽어, 인용된 내용이 해당 위치에 존재하는지 확인. 이 검증은 파일 읽기 연산만 필요하므로 LLM 호출 없이 수행 가능.

**Prompt Echo Ratio**:
```
ratio = findings_identical_to_review_target_description / total_findings
```
리뷰 요청의 원문 기술을 그대로 반복하는 발견사항은 실질적 분석 없이 요청을 앵무새처럼 반복하는 것으로 판단.

#### 6.5.2 Tier 2 저비용 신호 (청구항 4(b))

3개의 텍스트 통계 신호는 보고서 텍스트의 간단한 통계 분석으로 추출된다:

**Cross-Agent Agreement**:
```
agreement = matched_findings / total_findings (for this reviewer)
```
발견사항 매칭(6.3.2.1절) 결과를 활용하므로 추가 연산 비용 최소. 라운드 1에서는 비교 대상이 없으므로 1.0으로 초기화.

**Novelty Ratio** (라운드 2 이상):
```
novelty = new_findings_not_in_previous_round / total_findings
```
이전 라운드에 이미 존재했던 발견사항을 반복하는 비율. 라운드 1에서는 1.0으로 초기화.

**Hedge Density**:
```
density = count_of_hedge_phrases / (total_words_in_findings / 100)
```
회피적 표현: "it's worth noting", "arguably", "consider", "might", "potentially", "perhaps" 등.

#### 6.5.3 이원적 정규화 (청구항 4(c))

7개 신호를 0.0-1.0 범위로 정규화하되, 신호의 특성에 따라 두 가지 정규화 방식을 적용한다:

**비율 신호** (값이 클수록 나쁜 신호): Claim-to-Read, Prompt Echo, Hedge Density
```
normalized = 1.0 - min(value / threshold, 1.0)
```
예: Claim-to-Read 비율이 6.0이고 임계값이 3.0이면, `1.0 - min(6.0/3.0, 1.0) = 1.0 - 1.0 = 0.0` (최저점)

**비례 신호** (값이 클수록 좋은 신호): Specificity, Verifiability, Cross-Agent Agreement, Novelty
```
normalized = value (원래 값 그대로 사용, 이미 0-1 범위)
```

이원적 정규화의 기술적 효과: 신호의 방향성(높을수록 좋은 신호 vs. 높을수록 나쁜 신호)을 통일하여, 가중합으로 합성 점수를 산출할 때 모든 신호가 동일 방향(높을수록 건강)으로 기여하도록 한다.

#### 6.5.4 합성 건강 점수 (청구항 4(d))

```
Score = (
  norm(Claim-to-Read)   x 0.15 +
  Specificity            x 0.15 +
  Verifiability          x 0.25 +
  norm(Prompt Echo)      x 0.10 +
  norm(Hedge Density)    x 0.10 +
  Cross-Agreement        x 0.15 +
  Novelty                x 0.10
) x 100
```

Verifiability에 가장 높은 가중치(0.25)를 부여하는 이유: 인용된 코드 참조의 실제 존재 여부는 가장 객관적이고 검증 가능한 품질 신호이며, 환각(hallucination)의 가장 직접적인 지표이기 때문이다.

점수 기반 조치:
- 50 이상: 정상 (조치 없음)
- 30-49: AGENT_HEALTH_WARNING 로깅 + 신호 분석 기록, 계속 진행
- 30 미만: AGENT_HEALTH_FLAGGED, 프로필 설정에 따라 자동 퇴출 자격

#### 6.5.5 라운드 1 초기화 (청구항 4(e))

Cross-Agent Agreement와 Novelty Ratio는 이전 라운드 데이터가 필요하므로, 라운드 1에서는 비교 기준선이 존재하지 않는다. 이를 해결하기 위해 두 신호를 1.0(최고값)으로 초기화한다.

기술적 목적: 기준선 부재에 의한 오탐(false positive) 방지. 라운드 1에서 에이전트가 아직 교차 비교 기회를 갖지 못했음에도 낮은 Agreement/Novelty로 인해 부당하게 품질 저하로 판정되는 것을 방지한다.

### 6.6 상세 동작 — 종속항 D (구조적 제약 페르소나 기반 검증)

#### 6.6.1 설계 원리: AI 도움편향 우회

LLM 기반 AI 에이전트는 강한 도움편향(helpfulness bias)을 가지고 있어, "저품질 리뷰를 생성하라"는 직접적 지시를 거부하거나, 명시적으로 저품질 패턴을 생성하지 못한다. 본 발명의 테스트 페르소나는 이 문제를 우회하기 위해, 저품질을 직접 지시하지 않고 **구조적 제약**을 부여하여 간접적으로 저품질 패턴을 유도한다.

#### 6.6.2 Domain Amnesiac 페르소나 (청구항 5(a))

리뷰 대상의 실제 도메인과 전혀 무관한 전문성을 부여한다.

구현 실시예(참조: `.agents/prompts/bad-reviewer-a-domain-amnesiac.md`): Rust 시스템 코드를 리뷰 대상으로 하면서, 에이전트에게 "Web Accessibility & CSS Expert" 역할과 "WCAG 2.1 COMPLIANCE AUDIT" 전략을 부여한다. 에이전트는 ARIA 속성, 색상 대비, 키보드 네비게이션 등 웹 접근성 문제를 성실하게 찾으려 하지만, 대상이 Rust 시스템 코드이므로 모든 발견사항이 도메인 불일치로 판정된다.

트리거되는 품질 신호: 낮은 Cross-Agent Agreement (타 리뷰어의 발견사항과 매칭되지 않음), 낮은 Verifiability Score (접근성 관련 코드 참조가 존재하지 않음).

#### 6.6.3 Scope Constrained 페르소나 (청구항 5(a))

분석 범위를 극단적으로 제한한다.

구현 실시예(참조: `.agents/prompts/bad-reviewer-b-scope-constrained.md`): "Documentation Policy Reviewer" 역할로, 파일의 1-20번 라인만 분석하도록 지시한다. 모든 발견사항이 라인 1-20 내에 국한되어, 파일의 실질적 로직을 전혀 검토하지 못한다.

트리거되는 품질 신호: 낮은 Specificity Score (극소수 라인만 참조), 높은 Prompt Echo Ratio (제한된 범위에서 반복적 발견), 낮은 Cross-Agent Agreement.

#### 6.6.4 Overcautious 페르소나 (청구항 5(a))

과도한 위협 모델을 부여한다.

구현 실시예(참조: `.agents/prompts/bad-reviewer-c-overcautious.md`): "Nation-State Threat Assessment Reviewer" 역할로, 물리적 접근, 제로데이, 공급망 공격, 전자기 방사 분석, 양자 내성 암호화 등 국가 수준의 위협을 가정한다. 모든 발견사항을 CRITICAL 또는 HIGH로 분류하도록 지시한다.

트리거되는 품질 신호: 높은 Claim-to-Read 비율 (모든 것을 위협으로 판단), 낮은 Cross-Agent Agreement (과잉 경고로 타 리뷰어와 불일치), 높은 Hedge Density (위협 가능성을 광범위하게 기술).

#### 6.6.5 탐지율 검증 매트릭스 (청구항 5(b))

| 페르소나 | Claim-to-Read | Specificity | Verifiability | Prompt Echo | Hedge | Cross-Agree | Novelty |
|---------|:-----------:|:-----------:|:------------:|:-----------:|:-----:|:-----------:|:-------:|
| Domain Amnesiac | - | - | LOW | - | - | LOW | - |
| Scope Constrained | - | LOW | - | HIGH | - | LOW | - |
| Overcautious | HIGH | - | - | - | HIGH | LOW | - |

각 페르소나가 트리거하는 신호 패턴이 상이하므로, 품질 판별 시스템이 다양한 유형의 저품질 패턴을 탐지할 수 있는지 종합적으로 검증할 수 있다.

### 6.7 모델 계층화 (청구항 8)

본 발명은 멀티에이전트 검증의 각 단계에서 요구되는 모델 능력 수준이 상이함을 인식하고, 비용 효율성과 판단 정확도의 균형을 위해 3계층 모델 정책을 적용한다:

| 계층 | 용도 | 모델 수준 | 근거 |
|------|------|----------|------|
| Generation | 보고서 초안, 요약 | 경량 모델 (예: Haiku) | 생성 작업은 상대적으로 낮은 판단력 요구 |
| Judgment | Phase 1 독립 리뷰, Phase 2 투표 | 중급 모델 (예: Sonnet) | 판단 작업에서 경량 모델은 높은 편향을 보임 |
| Arbitration | Phase 3 교차 심문 | 고급 모델 (예: Opus) | 논쟁 해결은 가장 높은 추론 능력 요구 |

### 6.8 이벤트 로깅 (청구항 10)

모든 단계에서 발생하는 이벤트는 JSONL(JSON Lines) 형식의 추가 전용(append-only) 로그에 기록된다. 이벤트 유형별 최소 필수 필드:

```json
{"type":"REVIEW_STARTED","review_id":"cr-20260328-143022","profile":"code-review","target":"...","timestamp":"2026-03-28T14:30:22Z"}
{"type":"REVIEWER_SPAWNED","review_id":"cr-20260328-143022","reviewer_id":"security","expertise":"Security Expert","model":"sonnet","timestamp":"..."}
{"type":"HEALTH_SCORE","review_id":"cr-20260328-143022","round":1,"reviewer_id":"security","score":72,"signals":{"claim_to_read":0.8,"specificity":0.9,"verifiability":0.85,"prompt_echo":0.95,"hedge_density":0.7,"cross_agreement":1.0,"novelty":1.0}}
{"type":"FINDING_DISMISSED","review_id":"cr-20260328-143022","round":1,"finding_id":"f3","severity":"MEDIUM","reason":"auto-dismiss (solo, R>=3)","strike_incremented":false,"domain_consistent":true}
{"type":"AGENT_DISMISSED","review_id":"cr-20260328-143022","target_id":"reviewer-a","votes":[{"voter":"orchestrator","vote":"dismiss"}],"reason":"strike_count 2 >= threshold 2","replacement_id":"reviewer-a-prime"}
```

이벤트 로그의 기술적 효과:
- 사후 감사(post-hoc audit): 모든 투표, 기각, 퇴출 결정의 근거를 추적 가능
- 투표 오류 분석: CONTESTED→CONFIRMED 전환 이력으로 합의 전략의 적절성 평가
- 건강 점수 추세: 라운드별 건강 점수 변화로 에이전트 품질 저하 추세 감지

### 6.9 프로필 기반 구성 (청구항 9)

프로필 시스템은 다양한 검증 시나리오에 맞춤 구성을 가능하게 한다.

구현 실시예에서의 프로필 예시:

**코드 리뷰 프로필** (3 리뷰 에이전트, 전원합의, 2 클린 라운드):
- 보안 전문가: 패턴 기반 보안 스캔
- 정확성 분석가: 라인별 로직 분석
- SLOP 탐지기: 주장 검증 + 품질 신호 계산

**보안 리뷰 프로필** (4 리뷰 에이전트, 전원합의, 3 클린 라운드):
- 무관용(zero tolerance) 정책으로 더 높은 확인 기준 적용

**문서 리뷰 프로필** (2 리뷰 에이전트, 경량):
- R=2 가드 자동 적용
- 공식기각 비활성화

프로필 상속 메커니즘: `metadata.extends: _base`로 기반 프로필을 상속하고, 자식 프로필에서 필요한 파라미터만 오버라이드. 심층 병합(deep merge)으로 중첩된 객체의 부분 오버라이드 지원.

### 6.10 실시예의 실험 결과

본 발명의 구현 실시예에 대해, 의도적으로 주입된 결함 파일 및 정상 파일을 대상으로 복수의 검증 실험을 수행하여 각 구성요소의 효과를 정량적으로 확인하였다. 이하 실험 결과는 상기 구현 실시예의 실측 데이터에 기반한다.

#### 6.10.1 탐지율 검증 (Phase 1-2 프로토콜)

SQL injection 취약점이 의도적으로 주입된 테스트 파일(`bug-05-sql-injection.rs`)에 대해, 3명의 독립 리뷰 에이전트(보안, 정확성, SLOP탐지)를 스폰하여 검증한 결과:

| 대상 | 예상 | 실제 | 결과 |
|------|------|------|------|
| SQL injection (CRITICAL) | 탐지 | 3/3 unanimous CONFIRMED | PASS |

3명의 리뷰 에이전트가 격리된 컨텍스트에서 독립적으로 동일한 CRITICAL 취약점을 탐지하였으며, 발견사항 매칭 알고리즘에 의해 자동으로 그룹화되어 전원합의(unanimous)로 확인(CONFIRMED)되었다. 이는 단일 에이전트에 의한 리뷰보다 높은 탐지 신뢰성을 보여준다.

#### 6.10.2 도메인 인지 스트라이크 면제 검증

동일 테스트(bug-05-sql-injection.rs)에서, Overcautious 페르소나(nation-state 위협 모델)를 3번째 리뷰어로 투입한 실험 결과:

| 발견사항 | 정확성 | 보안 | Overcautious | 분류 |
|---------|:-----:|:---:|:----------:|------|
| SQL injection | CRITICAL | CRITICAL | CRITICAL | **CONFIRMED (3/3)** |
| Error info leakage | - | HIGH | HIGH | **CONTESTED (2/3)** |
| Silent row error | LOW | LOW | HIGH | **CONFIRMED (3/3)** |
| Audit log self-referential | - | - | CRITICAL | **AUTO-DISMISSED** |
| No caller authorization | - | - | CRITICAL | **AUTO-DISMISSED** |
| Offset unbounded DoS | - | - | HIGH | **AUTO-DISMISSED** |
| 기타 3건 | - | - | HIGH | **AUTO-DISMISSED** |

Overcautious 페르소나는 5건의 자동기각 발견사항으로 인해 5 스트라이크가 누적되어 AGENT_HEALTH_FLAGGED로 판정되었다. **핵심 불변식 검증**: SQL injection(3/3 CONFIRMED)과 silent row error(3/3 CONFIRMED)는 동일 리뷰어가 제출했더라도, 확인된 발견사항이므로 스트라이크에 기여하지 않았다. 즉, 무죄 증거(exculpatory) 불변식이 정상 작동함을 실증하였다.

#### 6.10.3 퇴출 및 안티앵커링 교체 검증

정상 파일(`constants.ts`)에 대해 Overcautious 페르소나를 투입한 실험:

1. Overcautious 페르소나: timing oracle + autosave side-channel 2건 자동기각 → 2 스트라이크 ≥ 임계값 2 → **AGENT_HEALTH_FLAGGED**
2. 기각 분석: 두 발견사항 모두 nation-state 수준에서만 유효, 정상 리뷰어 0명 동의
3. 퇴출 결정: **AGENT_DISMISSED**
4. 교체 리뷰어 A' 스폰: 대상 파일(constants.ts)만 전달, 선행 기각 이력 차단

**안티앵커링 효과 검증**:

| 퇴출된 에이전트의 기각 발견사항 | 교체 에이전트 A'가 재현? |
|-------------------------------|:---------------------:|
| Timing oracle (thresholds) | **NO** |
| Autosave side-channel | **NO** |

교체 에이전트 A'는 정상 리뷰어와 동일한 문제(dead code, path leak)를 독립적으로 발견하였으며, 선행 에이전트의 기각된 발견사항(timing oracle, autosave side-channel)을 재현하지 않았다. 이는 안티앵커링 정보 차단이 편향 전파를 효과적으로 방지함을 실증한다.

#### 6.10.4 제로비용 품질 신호 검증 (Verifiability Score)

미리 작성된 악의적 보고서(malformed report)로 Verifiability Score를 검증한 실험:

- 테스트 보고서: `bug-01-race-condition.rs`에 대해 3건의 발견사항을 기재하되, 라인 5, 10, 15를 인용
  - F-1: "Race condition in child-wait thread" → 라인 5 실제 내용: `use std::sync::{Arc, Mutex};` → **불일치**
  - F-2: "Registry removal not atomic" → 라인 10 실제 내용: `pub(crate) struct PtyHandle {` → **불일치**
  - F-3: "Missing Drop implementation" → 라인 15 실제 내용: `pub type PtyRegistry = ...` → **불일치**

**Verifiability Score: 0/3 = 0.00** (임계값 0.70 미만 → 저품질 신호 발동)

동일 대상에 대해 Scope Constrained 페르소나를 실제 투입한 결과, AI의 도움편향(helpfulness bias)으로 인해 제한된 범위(1-20번 라인) 내에서도 정확한 인용의 발견사항을 생성하였고, Verifiability Score는 5/5 = 1.00으로 오탐(false positive)이 발생하지 않았다. 이는 Verifiability Score가 실제 환각(hallucination)에 대해서만 반응하고, 범위 제한 리뷰어에 대해서는 오탐을 생성하지 않는 선택적 탐지 능력을 보여준다.

#### 6.10.5 No False Dismissal 검증 (스트라이크 임계값 경계값 테스트)

`bug-03-integer-overflow.rs`에 대해 Overcautious 페르소나를 임계값 3으로 투입한 실험:

| 발견사항 | 정확성 | 보안 | Overcautious | 분류 |
|---------|:-----:|:---:|:----------:|------|
| Unit mismatch (millis vs secs) | CRITICAL | CRITICAL | CRITICAL | **CONFIRMED (3/3)** |
| u128→u64 truncation | MINOR | HIGH | CRITICAL | **CONFIRMED (3/3)** |
| unwrap_or(0) clock rollback | - | - | CRITICAL | **AUTO-DISMISSED** |
| Timing side-channel | - | - | HIGH | **AUTO-DISMISSED** |

Overcautious 페르소나: 2 스트라이크 < 3 임계값 → **퇴출 안 됨**. 확인된 2건(unit mismatch, truncation)은 보존됨. 이는 임계값 경계 조건에서 부당 퇴출(false dismissal)이 방지됨을 실증한다.

## 7. 발명의 효과

### 7.1 컴퓨터 기능 자체의 개선

본 발명은 범용 컴퓨터에서 실행되는 멀티에이전트 소프트웨어 검증 시스템의 기능 자체를 개선하는 구체적이고 기술적인 해결책을 제공한다:

#### 7.1.1 제로비용 품질 판별에 의한 연산 자원 절감

기존 기술은 에이전트 품질 평가를 위해 추가 LLM 추론 호출(예: 별도의 평가 에이전트 스폰, 임베딩 기반 유사도 계산)을 필요로 하여 상당한 GPU 연산 자원과 API 비용을 소비한다. 본 발명은 리뷰 에이전트의 구조화된 보고서 메타데이터(파일 읽기 목록, 파일:라인 참조, 텍스트 통계)로부터 7개 품질 신호를 추출하여 합성 건강 점수를 산출함으로써, 추가 LLM 호출 없이(제로비용으로) 에이전트 품질을 판별한다. 이는 N명의 리뷰 에이전트에 대해 N회의 추가 평가 호출을 절감하는 효과가 있다.

#### 7.1.2 도메인 인지에 의한 전문가 에이전트 보호

기존 기술은 도메인 구분 없이 발견사항 기각을 처리하므로, 보안 전문가가 다른 비보안 에이전트에 의해 부당하게 퇴출되는 문제가 발생한다. 본 발명의 도메인 일관성 검사는 기각된 발견사항의 도메인이 해당 에이전트의 전문 도메인과 일치하는 경우 스트라이크를 면제하여, 전문적 시각에서의 정당한 발견사항이 다수결에 의해 묻히는 것을 방지한다.

#### 7.1.3 안티앵커링에 의한 검증 독립성 유지

퇴출된 에이전트의 이력이 교체 에이전트에 전달되면, 교체 에이전트가 선행 에이전트의 실패 패턴을 학습하여 유사 영역의 분석을 회피하는 부정적 앵커링(negative anchoring)이 발생한다. 본 발명은 확인된 발견사항만을 선택적으로 전달하고 기각/논쟁 이력을 차단함으로써, 교체 에이전트의 독립적 판단을 보장한다.

#### 7.1.4 이중소스 스트라이크 병합에 의한 통합적 품질 추적

자동기각(Phase 2)과 공식기각(Phase 3)은 서로 다른 메커니즘에 의해 발생하지만, 동일한 에이전트의 품질 저하를 나타낸다. 본 발명은 두 경로의 스트라이크를 단일 카운터에 병합하여, 하나의 경로에서만 임계값 미만이더라도 다른 경로와 합산하여 품질 저하 에이전트를 적시에 퇴출할 수 있다.

#### 7.1.5 양방향 컨텍스트 드리프트 감지

기존 기술(Agent Drift)은 에이전트 자신의 행동 드리프트만 측정한다. 본 발명은 리뷰어→요청자 방향의 교차 드리프트 감지를 구현하여, 요청자의 분석이 산출물의 현재 상태와 불일치하는 경우 검증 프로세스를 선택적으로 중단하고 복구함으로써, 무효 컨텍스트에 기반한 잘못된 검증을 방지한다.

#### 7.1.6 R=2 가드에 의한 무한루프 방지

2인 리뷰 구성에서 공식기각을 허용하면, 비제출자가 유일한 투표자로서 항상 기각할 수 있어 확인→기각→확인의 무한루프가 발생한다. 본 발명의 R=2 가드는 이 엣지 케이스를 구조적으로 차단하여, 시스템의 종료 보장(termination guarantee)을 제공한다.

### 7.2 조합의 신규성

상기 효과들은 개별 구성요소만으로는 달성할 수 없으며, 도메인 인지 스트라이크 + 이중소스 병합 + 안티앵커링 교체 + 제로비용 품질 판별 + 컨텍스트 드리프트 감지 + R=2/전원 가드의 6가지 기술적 구성요소가 결합되어 하나의 통합된 파이프라인으로 작동할 때 비로소 달성된다. 특히, 도메인 인지가 안티앵커링 교체와 결합됨으로써 "전문가 보호 + 편향 차단"이라는 양립하기 어려운 두 목표를 동시에 달성할 수 있다.

```

       │
       ▼

  │ Phase 1: 독립 리뷰       │

  │ 각각 격리된 컨텍스트      │

             │ R개 보고서 수집
             ▼
  ┌─────────────────────────┐

  │ 2. 분류                   │
  │    CONFIRMED / CONTESTED  │
  │    / AUTO-DISMISSED       │
  │ 3. 도메인 일관성 검사      │
  │ 4. 스트라이크 누적/면제    │
  │ 5. 건강 점수 산출          │
  └──────────┬──────────────┘
             │ CONTESTED 존재?
             ├─── No ──→ [라운드 결과]
             │
             ▼ Yes
  ┌─────────────────────────┐
  │ Phase 3: 선택적 토론      │
  │ 1. 교차심문 에이전트 스폰  │
  │    (arbitration 모델)     │
  │ 2. 최대 3 서브라운드       │
  │ 3. 최종 투표               │

  └──────────┬──────────────┘
             │
             ▼
  ┌─────────────────────────┐
  │ 라운드 결과               │
  │ 1. 드리프트 검사           │

             │
             ▼
```

도면 2는 각 라운드 내에서 Phase 1(독립 리뷰), Phase 2(투표), Phase 3(선택적 토론), 라운드 결과의 순서로 진행되는 3-Phase 흐름을 도시한다.

### 요약
